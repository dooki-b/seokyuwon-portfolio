## 2. 작업 원칙 (반드시 준수)

### ✅ 지향
1. **디자인 시스템 일관성**: 컬러·폰트·간격은 정의된 토큰 사용. 새 토큰 추가 금지.
2. **시맨틱 HTML**: `<section>`, `<article>`, `<header>` 정확히 사용. div 남용 금지.
3. **접근성**: aria-label, alt 속성, 키보드 네비게이션, 충분한 대비.
4. **모션 절제**: Framer Motion variants는 `app/lib/motion.ts`에서만 사용. 새 모션 추가 시 거기에 박을 것.
5. **반응형**: 모바일 우선 (`sm:`, `md:`, `lg:` Tailwind 브레이크포인트).
6. **TypeScript 0 에러**: `tsc --noEmit` 통과 필수.

### ❌ 회피
1. **AI 클리셰 디자인 금지**:

   - ❌ 부드러운 그림자 + 둥근 카드 그리드 클리셰

3. **콘텐츠 자의 변경 금지**:

## 3. 작업 지시 형식

작업 요청은 다음 형식으로 받습니다:

```
요청: [무엇을 추가/수정할지]
위치: [어떤 파일/섹션]
참고: [추가 컨텍스트, 있으면]
```

응답 형식:
1. **계획**: 무엇을 어떻게 할지 1~3문장 요약
2. **파일 변경 목록**: 추가/수정할 파일 명시
3. **코드**: 각 파일별 전체 또는 변경 부분 (diff 형식 권장)
4. **검증**: TS 에러 0건 + 빌드 통과 여부 확인 절차
5. **알려진 한계**: 작업 후 남은 우려사항이 있으면 명시

## 4. 자주 받을 작업 시나리오

### 시나리오 A: 새 섹션 추가
예: "에세이 섹션을 Strengths 다음에 추가해줘"
1. `app/components/` 새 컴포넌트 파일 생성
2. 기존 컴포넌트 패턴 따라 작성 (motion variants, 그리드, 타이포 토큰 동일)
3. `app/page.tsx`에 import + 적절한 위치 삽입
4. Header.tsx의 NAV_ITEMS에 추가

### 시나리오 B: 시각 자료 추가
예: "CookDex 스크린샷 4장을 Project 섹션에 추가해줘"
1. `public/screenshots/` 폴더 생성
2. `next/image`의 `<Image>` 컴포넌트 사용 (자동 최적화)
3. `Project.tsx`에 갤러리 그리드 또는 캐러셀 추가
4. alt 속성 정확히 작성

### 시나리오 C: 다국어 지원
예: "한·영 다국어 추가해줘"
1. `next-intl` 라이브러리 추가
2. `messages/ko.json`, `messages/en.json` 생성
3. 모든 컴포넌트의 텍스트를 키로 추출
4. 헤더에 언어 토글 추가

### 시나리오 D: 다크 모드 토글 (선택)
예: "다크 모드 토글 추가해줘"
1. `next-themes` 라이브러리 추가
2. 다크 모드 컬러 시스템 정의 (현재 라이트 미학과 별도)
3. 토글 버튼 헤더에 추가
4. **중요**: 라이트 모드가 기본이며 편집 미학 핵심임 — 다크는 옵션

### 시나리오 E: 블로그 / 에세이 게재
예: "블로그 페이지 추가, MDX 지원"
1. `@next/mdx` + `contentlayer` 추가
2. `app/blog/[slug]/page.tsx` 동적 라우트
3. 메인 페이지에서 Strengths 다음에 "Latest Essay" 카드 표시
4. 에세이 마크다운 → MDX 변환 가이드

## ⚠️ 솔직 안내 — AI 에이전트 사용 시 주의

### 1. 사실 변경 위험
AI가 "이게 더 자연스럽다"고 판단하고 본인 정보를 바꿀 수 있습니다.
→ **변경 후 항상 본인 정보·수치 대조 검증 필수**

### 2. 디자인 시스템 일탈
AI가 다른 색·폰트를 임의 추가할 수 있습니다.
→ **위 프롬프트의 "회피" 섹션을 매번 강조 권장**

### 3. 의존성 폭주
AI가 매번 새 라이브러리 추가 제안할 수 있습니다.
→ **꼭 필요한 경우에만 승인. package.json 변경은 신중히**

### 4. 모션 과다
AI가 매 컴포넌트마다 다른 모션을 추가하려 할 수 있습니다.
→ **motion.ts variants 재사용 강제. 새 variant 추가는 의도적으로만**

### 5. 빌드 검증
모든 변경 후 본인이 직접:
```
npm run build
```
실행하여 TS 0 에러 + 빌드 통과 확인.
